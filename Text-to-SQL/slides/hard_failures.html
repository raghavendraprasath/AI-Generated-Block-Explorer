When Text-to-SQL Fails

Three hard cases where incorrect SQL still returns a plausible wrong answer

Reference: Spider 2.0 leaderboard — complex SQL beyond current model limits

← → arrow keys  ·  F11 fullscreen

Case 1 — Column selection

What is the hash of the block at the maximum height?
Correct SQLSELECT hash FROM blocks ORDER BY height DESC LIMIT 1
Correct answer0000000000001c1a…f681b39
LLM SQLSELECT blocks.height FROM blocks WHERE height = (SELECT MAX(height) FROM blocks)
Wrong answer110028 (height, not hash)

The model returns a valid number but answers the wrong column — hash vs height.

Case 2 — Aggregation logic

Which block height has the most transactions?
Correct SQLSELECT block_height FROM transactions GROUP BY block_height ORDER BY COUNT(*) DESC LIMIT 1
Correct answer93107
LLM SQLSELECT MAX(block_height) FROM transactions
Wrong answer110028

MAX(block_height) = highest block with any tx ≠ block with the largest tx count.

Case 3 — Domain enums

What is the total BTC value of all pubkeyhash (P2PKH) outputs?
Correct SQL… WHERE scriptpubkey_type = 'pubkeyhash'
Correct answer25,913,494.67 BTC
LLM SQL… WHERE scriptpubkey_type = 'pubkey'
Wrong answer5,989,823.01 BTC

P2PK (pubkey) and P2PKH (pubkeyhash) are different script types in Bitcoin Core.

Takeaway

  • Schema in context is not enough for free LLMs
  • Failures: wrong column, wrong aggregation, wrong enum
  • Wrong SQL can still execute and look plausible
  • Golden SQL tests (12/12) validate the database; live LLM is the weak link
